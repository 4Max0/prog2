## Aufgabe: Git Status erklären

```bash
pm-lecture % git status
On branch b03

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   CONTRIBUTING.md
        modified:   homework/b03.md

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        foo.java

no changes added to commit (use "git add" and/or "git commit -a")
```


Der Befehl git status ist ein Befehl von git der den Status einer ausgewählten Codebase-branch anzeigt. 
Es wird angezeigt welche dateien modifiziert, neu hinzugefügt oder gelöscht wurden, seit dem letzten 
commit der Codebase.

In dem Fall des Beispiels ist es die "branch b03", die wir uns anschauen. Die Ausgabe verät uns des 
weiteren das der Nutzer die Dateien `CONTRIBUTING.md` und `homework/b03.md` bearbeitet hat.

Es wird zudem darauf hingwiesen, dass noch keine der Dateien, die modifiziert, hinzugefügt oder gelöscht 
wurden, per `git add <file>` zu dem nächsten commit hinzugefügt wurden. Ebenso weist git darauf hin dass 
`git restore <file>` benutzt werden kann um, eine datei wieder zu ihrer ursprünglichen Form zu führen.

Wenn wir jetzt die Datei foo.java hinzufügen wollen, so dass die gemachten Änderungen in den nächsten commit 
berücksichtigt werden, geben wir einfach `git add <file>` ein für die Datei:

```bash
git add foo.java
```

Mit den obigen Befehl wird nur die Datei `foo.java` in den nächsten commit berücksichtigt. Wenn man dagegen 
alle dateien hinzufügen will, kann man dass durch `git add .` oder `git commit -a` (wodurch der commit 
direkt durchgeführt wird, was fehler bergen kann). 

Um dann zu commiten kann man dann z.B. folgenden Befehl eingeben:

```bash
git commit -m "Commit for foo.java"
```

Q: Wie finden Sie auf der Konsole heraus, welche Tasks es gibt?
A: Die möglichen Tasks können per `gradle tasks` angesehen werden.

Q: Erklären Sie das Projektlayout, d.h. wo kommen beispielsweise die Java-Dateien hin?
A: Die Java Dateien kommen bei einem Projekt in das passende Modul z.B. das haupt modul
`app/src/main/java/` und tests kommen in `app/src/test/java` die main Klasse ist dabei automatisch bei 
`app/src/main/java/org/example/app/App.java`

Q: Erklären Sie, in welche Abschnitte das generierte Buildskript unterteilt ist und welche Aufgaben diese 
Abschnitte jeweils erfüllen. Gehen Sie dabei im Detail auf das Plugin application und die dort 
bereitgestellten Tasks und deren Abhängigkeiten untereinander ein.
A: Die buidl.gradle hat folgendes:
- `plugins`: aktivieren von plugins 
    - Modulare erzeugung eines Projects
    - wiederverwendbare struktur
- `repositories`: wo gradle bibliotheken her bekommt (Maven Central z.B.)
- `dependencies`: hinzufügen welche bibliotheken genutzt werden sollen
- `java`: einstellen von eigenschaften (z.B. Main location)
- `application`: Tasks konfigurieren
Das plugin `application`  macht dabei folgendes:
- beschreibt das projekt als eine Java Applikation
- Macht es ausführbar
- erstellt standard-tasks wie run
- erlaubt das runnen von Applikationen in der JVM

Q: Öffnen Sie das Projekt in Ihrer IDE. Wie können Sie hier die verschiedenen Tasks ansteuern?
A: In JetBrains gibt es rechts den Elefanten-Button der für Gradle zuständig ist (mit dem plugin).
